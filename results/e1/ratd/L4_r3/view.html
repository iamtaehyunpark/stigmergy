# Research Notes: Short-term/Context Window Memory

## 1. Definition and Scope
Short-term memory in LLM agents refers to the immediate context available to the model during inference. Unlike long-term memory (vector stores, databases), this is transient, bounded by the model's context window, and directly influences the current generation.

## 2. Key Mechanisms

### A. Context Window
- **Definition**: The maximum number of tokens (input + output) an LLM can process in a single forward pass.
- **Current Standards**: Ranges from 4k (older models) to 128k+ (modern models like GPT-4, Claude 3, Gemini).
- **Limitation**: Fixed upper bound; exceeding it requires truncation or summarization.

### B. Attention Mechanism
- **Self-Attention**: Allows the model to weigh the importance of different tokens in the context window relative to each other.
- **Recency Bias**: Models often attend more strongly to recent tokens (end of context) than distant ones (beginning), a phenomenon known as "lost in the middle."

### C. Context Management Strategies
1. **Truncation**: Removing oldest tokens when the limit is reached. Simple but loses historical context.
2. **Summarization**: Compressing older dialogue turns into a summary to preserve key information while reducing token count.
3. **Sliding Window**: Keeping only the most recent N tokens.
4. **Key-Value Caching**: Storing attention states for previous tokens to speed up inference, though this is more of an optimization than a memory strategy per se.

## 3. Architectural Considerations
- **Statelessness**: LLMs are inherently stateless; short-term memory is constructed by the application layer (the agent framework) by prepending history to the prompt.
- **Tokenization**: Different models use different tokenizers (BPE, SentencePiece), affecting how "words" map to context slots. Punctuation and whitespace consume tokens.

## 4. Limitations and Challenges
- **Capacity Constraints**: Complex tasks may require more context than the window allows.
- **Noise Sensitivity**: Irrelevant information in the context window can degrade performance (distractor tokens).
- **Cost**: Longer contexts increase computational cost and latency.
- **Consistency**: Maintaining coherent persona or instructions across a long context window is difficult due to attention decay.

## 5. Terminology Survey (Preliminary)
- *Context Window*: Standard term.
- *Short-term Memory*: Often used interchangeably with "conversation history" or "prompt context."
- *Attention*: Core mechanism.
- *Token Limit*: Hard constraint.
- *Recency Bias*: Observed behavior.

## 6. References for Further Reading
- Vaswani et al., "Attention Is All You Need" (2017)
- OpenAI API Documentation on Context Windows
- Anthropic Research on Claude's Context Handling
- "Lost in the Middle: How Language Models Use Long Contexts" (Liu et al., 2023)

# Research Notes: Chapter 2 - Long-term/Vector Store Memory

## 1. Core Concepts & Definitions
- **Long-term Memory**: Persistent storage of information beyond the immediate context window, allowing agents to recall past interactions, facts, or skills over extended periods.
- **Vector Store (Vector Database)**: A specialized database designed to store, index, and retrieve high-dimensional vectors (embeddings) efficiently. Key operations include similarity search (e.g., cosine similarity, Euclidean distance).
- **Embeddings**: Dense vector representations of data (text, images, etc.) generated by neural networks (e.g., Sentence-BERT, OpenAI Embeddings). They capture semantic meaning, enabling semantic search.
- **Retrieval-Augmented Generation (RAG)**: A paradigm where an LLM retrieves relevant context from an external knowledge base (like a vector store) before generating a response, improving accuracy and reducing hallucinations.

## 2. Vector Store Architectures & Technologies
- **In-Memory vs. Disk-Based**: 
  - *In-Memory*: Fast retrieval, limited by RAM (e.g., FAISS, Chroma in-memory mode).
  - *Disk-Based*: Scalable, persistent, slightly slower (e.g., Pinecone, Weaviate, Milvus, Qdrant).
- **Indexing Strategies**:
  - *Brute Force*: Linear scan, accurate but slow for large datasets.
  - *Approximate Nearest Neighbor (ANN)*: Trade-off between speed and accuracy.
    - *HNSW (Hierarchical Navigable Small World)*: Graph-based, high recall, fast search. Widely used (e.g., in Milvus, Qdrant).
    - *IVF (Inverted File Index)*: Clusters vectors, searches within clusters. Good for large datasets.
    - *LSH (Locality-Sensitive Hashing)*: Hash-based, fast but lower recall.
- **Key Vendors/Libraries**:
  - *FAISS*: Facebook AI Similarity Search, research-focused, highly optimized.
  - *Chroma*: Developer-friendly, embedded, good for prototyping.
  - *Pinecone*: Fully managed cloud service, easy integration.
  - *Weaviate*: Hybrid search (vector + keyword), GraphQL API.
  - *Milvus*: Scalable, open-source, supports billion-scale vectors.
  - *Qdrant*: Rust-based, high performance, filterable vector search.

## 3. Embedding Models & Strategies
- **Model Selection**: 
  - *General Purpose*: OpenAI `text-embedding-ada-002`, BGE (BAAI General Embedding), E5.
  - *Domain-Specific*: Fine-tuned models for legal, medical, or technical domains.
- **Embedding Granularity**:
  - *Document-level*: Embed entire documents. Good for broad context.
  - *Chunk-level*: Split documents into smaller chunks (e.g., 500-1000 tokens). Improves retrieval precision.
  - *Sentence-level*: Embed individual sentences. High precision, higher storage cost.
- **Hybrid Embeddings**: Combining multiple models or modalities (e.g., text + metadata) for richer representations.

## 4. Retrieval Strategies & Optimization
- **Similarity Metrics**:
  - *Cosine Similarity*: Measures angle between vectors, insensitive to magnitude. Most common.
  - *Euclidean Distance*: Measures straight-line distance. Sensitive to magnitude.
  - *Dot Product*: Similar to cosine but considers magnitude. Used in some ANN libraries.
- **Retrieval Techniques**:
  - *Simple Vector Search*: Query embedding vs. stored embeddings.
  - *Hybrid Search*: Combines vector search with keyword (BM25) or metadata filtering. Improves recall for exact matches.
  - *Re-ranking*: After initial retrieval, use a cross-encoder model (e.g., Cohere Rerank, BGE Reranker) to re-score and reorder results for higher precision.
  - *Multi-Query Retrieval*: Generate multiple query variations from the user's input to retrieve diverse relevant chunks.
  - *Contextual Compression*: Retrieve chunks, then compress or summarize them to fit context window while retaining key info.
- **Metadata Filtering**: Use structured metadata (e.g., date, source, author) to filter results before or during vector search. Crucial for personalization and access control.

## 5. Memory Management & Lifecycle
- **Storage & Indexing**:
  - *Chunking Strategies*: Recursive character splitting, semantic chunking (using LLMs to identify boundaries), or fixed-size with overlap.
  - *Index Updates*: Handling new data (insertions) and outdated data (deletions/updates). Incremental indexing vs. full re-indexing.
- **Recall & Forgetting**:
  - *Active Recall*: Periodically query the vector store to reinforce important memories.
  - *Decay Mechanisms*: Reduce the weight or relevance of older memories over time, or explicitly delete irrelevant data.
  - *Consolidation*: Merge similar memories or summarize clusters of memories to save space and improve retrieval efficiency.
- **Scalability Challenges**:
  - *Latency*: Ensuring fast retrieval as the dataset grows.
  - *Cost*: Storage and compute costs for large-scale vector stores.
  - *Consistency*: Maintaining data integrity across distributed systems.

## 6. Integration with Agent Frameworks
- **LangChain**: Provides abstractions for vector stores (`VectorStore` interface), retrievers (`VectorStoreRetriever`), and memory (`VectorStoreRetrieverMemory`).
- **LlamaIndex**: Focuses on data indexing and retrieval, with advanced features like query pipelines and hybrid search.
- **Custom Implementations**: Direct integration with vector store APIs for fine-grained control.

## 7. Best Practices & Pitfalls
- **Best Practices**:
  - Choose the right chunk size and overlap for your use case.
  - Use hybrid search for better recall.
  - Implement re-ranking for higher precision.
  - Regularly evaluate retrieval quality (e.g., using MRR, NDCG).
  - Monitor and optimize index performance.
- **Pitfalls**:
  - *Semantic Drift*: Embeddings may not capture nuanced meanings over time.
  - *Cold Start*: Difficulty retrieving relevant information for new topics with few examples.
  - *Over-retrieval*: Retrieving too much irrelevant context, overwhelming the LLM.
  - *Under-retrieval*: Missing key information due to poor embedding quality or retrieval strategy.

## 8. Emerging Trends
- **Multimodal Vector Stores**: Storing and retrieving embeddings for text, images, audio, and video.
- **Graph-Vector Hybrids**: Combining vector stores with knowledge graphs for structured reasoning.
- **On-Device Vector Stores**: Lightweight vector databases for edge devices.
- **Self-Improving Memory**: Agents that automatically update and refine their memory based on feedback.

# Research Notes: Chapter 3 - Episodic/Experience Replay Memory

## 1. Core Concept Definition
Episodic memory in LLM agents refers to the storage and retrieval of specific past interactions, trajectories, or experiences. Unlike semantic memory (general knowledge) or working memory (current context), episodic memory allows agents to learn from history, avoid repeating mistakes, and generalize from similar past situations.

## 2. Key Architectural Patterns

### A. Experience Replay Buffers
- **Origin**: Reinforcement Learning (RL), specifically DQN (Deep Q-Networks).
- **Mechanism**: A fixed-size buffer stores tuples of (state, action, reward, next_state). During training, mini-batches are sampled uniformly or prioritized.
- **Adaptation for LLMs**: 
  - Tuples become (prompt, response, feedback/score, context).
  - Used for fine-tuning (RLHF/DPO) or online adaptation.
  - **Challenge**: LLM outputs are high-dimensional and non-stationary; standard RL replay assumptions often break.

### B. Trajectory Management
- **Definition**: Storing complete sequences of agent interactions (multi-turn dialogues, tool-use chains).
- **Structure**: 
  - Nodes: Individual steps (user query, agent thought, tool call, observation).
  - Edges: Temporal/causal links.
- **Retrieval**: 
  - Similarity search over trajectory embeddings.
  - Prefix matching for early-stage alignment.
  - Graph-based traversal for complex reasoning paths.

### C. Episodic Retrieval-Augmented Generation (RAG)
- **Mechanism**: Before generating a response, the agent retrieves relevant past episodes from a vector database.
- **Key Components**: 
  - Embedding model for episode representation.
  - Vector index (FAISS, Pinecone, etc.).
  - Re-ranking module for precision.
- **Use Case**: Personalized assistants, consistent character behavior, debugging past errors.

## 3. Critical Challenges & Solutions

### A. Catastrophic Forgetting
- **Problem**: New experiences overwrite or interfere with old ones.
- **Solutions**: 
  - Elastic Weight Consolidation (EWC) analogs for LLMs.
  - Replay-based regularization during fine-tuning.
  - Modular memory architectures (separate buffers for different domains).

### B. Scalability & Efficiency
- **Problem**: Storing full trajectories is expensive; retrieval latency increases with buffer size.
- **Solutions**: 
  - Summarization: Compress episodes into concise summaries.
  - Hierarchical memory: Short-term (working) vs. long-term (episodic) buffers.
  - Approximate Nearest Neighbor (ANN) search for fast retrieval.

### C. Noise & Irrelevance
- **Problem**: Not all past experiences are useful; some are misleading.
- **Solutions**: 
  - Prioritized replay: Weight experiences by reward/surprise.
  - Filtering: Remove low-quality or redundant episodes.
  - Confidence scoring: Only retrieve high-confidence matches.

## 4. Emerging Trends

### A. Self-Reflective Episodic Memory
- Agents explicitly reflect on past episodes to extract lessons.
- Example: "Why did this tool call fail?" -> Store the failure mode and correction.

### B. Multi-Agent Episodic Sharing
- Agents share experiences across a team.
- Requires consensus mechanisms and conflict resolution.

### C. Neuro-Symbolic Integration
- Combining neural episodic retrieval with symbolic reasoning over past events.
- Enables causal analysis of past trajectories.

## 5. Canonical Terminology Survey (Preliminary)
- **Episode**: A single interaction or trajectory segment.
- **Replay Buffer**: Storage for past experiences.
- **Trajectory**: A sequence of states/actions/observations.
- **Retrieval-Augmented Generation (RAG)**: Using retrieved episodes to guide generation.
- **Prioritized Replay**: Sampling experiences based on importance.
- **Catastrophic Forgetting**: Loss of old knowledge due to new learning.

## 6. References & Further Reading
- Mnih et al. (2015): Human-level control through deep reinforcement learning (DQN).
- Schaul et al. (2015): Prioritized Experience Replay.
- Lewis et al. (2020): Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks.
- Recent papers on LLM agent memory (e.g., MemGPT, AutoGPT memory modules).

# Research Notes: Chapter 4 - Semantic/Graph-based Memory

## 1. Core Concepts & Definitions
*   **Semantic Memory**: Long-term memory of meanings, concepts, and facts, distinct from episodic (event-based) memory. In LLM agents, this refers to structured knowledge that persists across sessions and provides context-independent grounding.
*   **Knowledge Graph (KG)**: A directed graph where nodes represent entities/concepts and edges represent relationships. It enables explicit reasoning and traversal.
*   **Vector Database (Vector DB)**: Stores high-dimensional embeddings. Enables semantic similarity search but lacks explicit structural relationships.
*   **GraphRAG**: A hybrid approach combining Retrieval-Augmented Generation (RAG) with Knowledge Graphs. It uses KGs to provide structured context and vector search for dense semantic matching.
*   **Ontology**: A formal representation of knowledge as a set of concepts within a domain and the relationships between those concepts. Crucial for schema definition in KGs.

## 2. Architectural Patterns
*   **Hybrid Memory Systems**: Most modern agents use a combination of:
    *   *Vector Store*: For fast, approximate semantic retrieval of unstructured text chunks.
    *   *Graph Store*: For precise, relational queries (e.g., "Who is the CEO of Company X?").
*   **Graph Construction Pipeline**:
    1.  *Entity Extraction*: Identify entities (NER) from text.
    2.  *Relation Extraction*: Identify relationships between entities.
    3.  *Graph Population*: Insert nodes/edges into a graph database (e.g., Neo4j, Amazon Neptune).
    4.  *Embedding Generation*: Create vector embeddings for nodes/edges to enable hybrid search.
*   **Query Execution**:
    *   *Cypher/Gremlin Queries*: For exact structural matches.
    *   *Vector Search on Nodes*: For semantic similarity of entities.
    *   *Multi-hop Reasoning*: Traversing the graph to connect distant concepts.

## 3. Key Technologies & Tools
*   **Graph Databases**: Neo4j (Cypher), Amazon Neptune, ArangoDB, TigerGraph.
*   **Vector Databases**: Pinecone, Weaviate (supports hybrid search), Milvus, Qdrant.
*   **LLM Frameworks with Graph Support**:
    *   *LangChain*: Has `GraphCypherQAChain` and `GraphQAChain`.
    *   *LlamaIndex*: Supports `KnowledgeGraphIndex` and `PropertyGraphIndex`.
    *   *Microsoft GraphRAG*: A specific implementation for community detection and summarization over large corpora.

## 4. Advantages over Pure Vector RAG
*   **Explicit Relationships**: Captures "A is related to B" explicitly, not just via embedding proximity.
*   **Multi-hop Reasoning**: Can answer questions requiring multiple steps of inference (e.g., "Find all customers who bought product X and also bought product Y").
*   **Hallucination Reduction**: Grounding answers in explicit graph structures reduces reliance on LLM's internal parametric knowledge.
*   **Explainability**: The path taken through the graph can be shown to the user as evidence.

## 5. Challenges & Limitations
*   **Construction Cost**: Extracting accurate entities and relations from text is computationally expensive and error-prone.
*   **Schema Rigidity**: Defining an ontology requires domain expertise. Dynamic schemas are hard to manage.
*   **Scalability**: Graph queries can become slow on very large graphs without proper indexing.
*   **Maintenance**: Keeping the graph up-to-date with new information requires continuous ingestion pipelines.

## 6. Terminology Survey (for Standards Track)
*   *Common Terms*: Knowledge Graph, Entity, Relation, Node, Edge, Ontology, Schema, Cypher, Gremlin, Vector Embedding, Hybrid Search, GraphRAG, Multi-hop Reasoning, Entity Resolution, Relation Extraction.
*   *Potential Drift Points*:
    *   "Semantic Memory" vs. "Knowledge Graph": Sometimes used interchangeably, but KG is a specific implementation of semantic memory.
    *   "Vector DB" vs. "Graph DB": Distinct technologies, but often combined.
    *   "RAG" vs. "GraphRAG": GraphRAG is a subset/extension of RAG.

## 7. Recommended Structure for Chapter 4
1.  **Introduction**: Defining Semantic Memory in Agents.
2.  **Knowledge Graphs 101**: Nodes, Edges, Ontologies.
3.  **Building the Graph**: Extraction, Population, and Maintenance.
4.  **Hybrid Retrieval**: Combining Vector and Graph Search.
5.  **Case Studies**: GraphRAG, LangChain/LlamaIndex implementations.
6.  **Challenges & Best Practices**: Schema design, scalability, accuracy.
7.  **Conclusion**: The future of structured memory in agents.
